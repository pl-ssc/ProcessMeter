# Руководство по встроенной BI-аналитике

В проекте **ProcessMeter** аналитика доступна прямо внутри приложения и не зависит от Metabase или другого внешнего BI-инструмента. Источник данных для аналитики — SQL-представления из файла `db/bi_views.sql`.

## Источники данных

### `view_bi_labor_distribution`

Основной источник для анализа трудозатрат и проектирования целевой операционной модели.

- `fte`: эквивалент полной занятости (`labor_hours / 165`)
- `labor_hours`: трудозатраты в часах
- `executor_type`: категория исполнителя
- `process_level_1-4`: иерархия процесса
- `system_name`: используемая ИТ-система

### `view_bi_respondent_stats`

Операционный слой для контроля хода опроса.

- `invitation_at`: время приглашения пользователя
- `first_answer_at`: время первого ответа
- `completion_at`: время завершения опроса
- `elapsed_days_since_invitation`: длительность прохождения в днях
- `active_filling_days`: активное время работы в днях
- `total_labor_hours`: суммарные трудозатраты пользователя
- `completion_percentage`: процент заполнения анкеты
- `is_survey_completed`: флаг завершения опроса

### `view_bi_process_efficiency`

Слой для поиска узких мест и неэффективностей.

- `avg_labor_hours`: среднее время на операцию
- `notes_count`: количество комментариев к операциям

### `view_bi_survey_progress_total`

Сводное представление по общему прогрессу кампании.

- `total_respondents`: общее число приглашённых респондентов
- `completed_respondents`: число завершивших опрос

## Где это используется

- отдельная страница аналитики во frontend
- API-маршруты `/api/analytics/*`
- экспорт и дополнительный SQL-анализ на стороне БД

## Эксплуатационные заметки

- любые изменения структуры аналитики начинайте с `db/bi_views.sql`
- при изменении состава полей проверьте совместимость frontend и backend API
- если в будущем понадобится внешний BI, подключать его следует к этим SQL views, а не напрямую к нормализованным таблицам
