# Инструкция по первому развертыванию (Deployment Guide)

Этот документ описывает пошаговый процесс развертывания приложения ProcessMeter с нуля.

## Предварительные требования

*   **Node.js** v24+
*   **PostgreSQL** 15+ (установленный и запущенный сервер)
*   **Docker** (опционально, для запуска через Compose)
*   Доступ к эталонной базе данных (источник справочников процессов)

---

## Шаг 1: Настройка окружения

```bash
cp .env.example .env
```

Откройте `.env` и установите следующие переменные:

| Переменная | Описание | Обязательна в prod |
|---|---|---|
| `TARGET_DATABASE_URL` | Строка подключения к рабочей БД | ✅ |
| `JWT_SECRET` | Секрет подписи JWT (min 32 символа) | ✅ |
| `APP_URL` | Публичный URL приложения, используется в ссылках email | ✅ |
| `ADMIN_USERNAME` | Email первого администратора | ✅ |
| `ADMIN_PASSWORD` | Пароль первого администратора | ✅ |
| `ADMIN_FULL_NAME` | Полное имя администратора | — |
| `ORG_NAME` | Название организации | — |
| `SOURCE_DATABASE_URL` | Строка подключения к эталонной БД (для импорта справочников) | — |

> **Важно:** При `NODE_ENV=production` сервер не запустится, если `JWT_SECRET` не задан или равен значению по умолчанию `change-me`.

---

## Шаг 2: Установка зависимостей

```bash
cd backend && npm install
cd ../frontend && npm install
cd ..
```

---

## Шаг 3: Инициализация базы данных

Миграции применяются **автоматически при запуске** `npm start` через `start.js` (используется `node-pg-migrate`).

Запустить вручную:
```bash
cd backend
npm run migrate:up
```

Миграции хранятся в `backend/migrations/` и версионируются. Каждый новый запуск применяет только незапущенные шаги.

---

## Шаг 4: Запуск приложения

### Docker Compose (рекомендуется для production)

```bash
docker compose up --build -d
```

Приложение будет доступно на порту, указанном в `PORT` (по умолчанию `3001`).

### Режим разработки (ручной)

Терминал 1 (Backend):
```bash
cd backend
npm run dev
```

Терминал 2 (Frontend):
```bash
cd frontend
npm run dev
```

Фронтенд доступен на `http://localhost:5173`, API — на `http://localhost:3001`.

---

## Шаг 5: Создание первого администратора

Если переменные `ADMIN_USERNAME`, `ADMIN_PASSWORD` и `ADMIN_FULL_NAME` заданы в `.env`, администратор создаётся автоматически при первом запуске сервера (функция `ensureAdminUser` в `db/init.js`).

> **Требование:** `ADMIN_USERNAME` должен быть валидным email-адресом — он используется как адрес получателя при отправке системных уведомлений.

---

## Шаг 6: Настройка SMTP (email-уведомления)

SMTP-настройки задаются в административном интерфейсе: **Настройки → SMTP**.

Там можно проверить соединение кнопкой «Тест подключения» перед сохранением.

Email используется для:
- Приглашения новых пользователей (установка пароля по ссылке)
- Сброса/смены пароля

> Ссылки в письмах формируются на основе `APP_URL`. Убедитесь, что это значение доступно получателям письма.

---

## Шаг 7: Импорт справочников процессов

Выполните через административный интерфейс: **Администрирование → Импорт**.

Или через API:
```bash
curl -X POST http://localhost:3001/api/admin/import \
  -H "X-Admin-Key: YOUR_ADMIN_API_KEY"
```

> **Важно:** Импорт очищает `user_answers` и создаёт новые пустые ответы. Выполняйте до того, как пользователи начнут вводить данные.

## Возможные проблемы

### Сервер не запускается, ошибка `[FATAL] Missing required environment variables`
Задайте `JWT_SECRET` и `TARGET_DATABASE_URL` в `.env` и убедитесь, что `NODE_ENV=production` не установлен при локальной разработке.

### Пользователи не получают письма
1. Зайдите в **Настройки → SMTP** и нажмите «Тест подключения».
2. Убедитесь, что `APP_URL` настроен на публичный адрес (не `localhost`).
3. Проверьте логи сервера на предмет ошибок `sendPasswordLinkEmail failed`.

### Операции не отображаются у пользователя
1. Убедитесь, что импорт справочников прошёл успешно (должны отображаться процессы в дереве).
2. Администратор должен выдать пользователю права в разделе «Пользователи».
3. После выдачи прав `user_answers` генерируются автоматически.

### Ошибки при ответах пользователей не создались
Таблица `user_answers` заполняется функцией `copy_operations_to_user_answers`. Она вызывается при:
- Выдаче/изменении прав доступа пользователя
- Импорте справочников
