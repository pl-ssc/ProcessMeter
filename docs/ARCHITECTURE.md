# Архитектура проекта ProcessMeter

Проект представляет собой классическое клиент-серверное веб-приложение (SPA + REST API) с использованием реляционной БД. Проект оформлен в виде монорепозитория, разделенного на две основные части: `frontend` и `backend`.

Система спроектирована с учетом ожидаемой нагрузки: до 100 одновременно работающих пользователей и до 400 пользователей всего. Такая нагрузка считается легкой (Low/Medium-load) и позволяет использовать максимально простую, монолитную и надежную архитектуру без преждевременной оптимизации.

## 1. Технологический стек

- **Фронтенд**: React 18, Vite, нативные ES-модули (без TypeScript). Стилизация с помощью Vanilla CSS. Для отображения сложных таблиц данных используется высокопроизводительная библиотека `@glideapps/glide-data-grid`.
- **Бэкенд**: Node.js (v24+, `"type": "module"`), Fastify (веб-фреймворк).
- **База данных**: PostgreSQL 15. Взаимодействие с БД осуществляется посредством нативного драйвера `pg` (написание "сырых" SQL запросов с параметризацией для защиты от инъекций, без тяжелых ORM).
- **Инфраструктура**: Docker Compose. Для production-окружения рекомендуется использование Nginx (как reverse-proxy и сервера для раздачи статического фронтенда).

## 2. Архитектурная схема развертывания (Мультитенантность)

Система предполагает развертывание на инфраструктуре заказчика (on-premise). При этом выделяется **Центральный узел** (Master Data) для управления эталонными справочниками компании-интегратора, и изолированные **Узлы Заказчиков** с собственными базами данных.

```text

├── backend/            # Серверная часть (Node.js + Fastify)
│   ├── src/            # Исходный код API (сервер, роуты, контроллеры)
│   └── migrations/     # SQL-скрипты направленных миграций БД

```

## 4. Взаимодействие компонентов

1. **Frontend (SPA)**:
   - В режиме разработки запускается через Vite (`npm run dev`).
   - В production собирается в статические файлы (`dist`) и раздается через веб-сервер (в идеале Nginx, для упрощения развертывания может раздаваться через Fastify, но с меньшей эффективностью).
   - Взаимодействует с бэкендом посредством REST API. Состояние аутентификации (JWT) хранится на клиенте и передается в заголовке `Authorization`.
2. **Backend (Fastify API)**:
   - Обрабатывает бизнес-логику и проверяет права доступа на основе JWT-токенов (`@fastify/jwt`). Является Stateless-сервером (не хранит сессий в памяти).
   - Работает с БД через пул соединений `pg.Pool`. Для 100 одновременных пользователей базового пула на 10-20 соединений более чем достаточно.
3. **Database (PostgreSQL)**:
   - **Управление справочниками (Master Data Management)**: Так как система разворачивается на серверах разных заказчиков, используется паттерн разделения баз.
     - **`refdb`** — централизованная (эталонная) база данных. Находится в контуре компании-разработчика/интегратора. Используется для хранения и валидации единых версий справочников (процессов, систем, исполнителей).
     - **`mydb`** — рабочая база данных. Находится в контуре Заказчика (on-premise). Хранит локальные копии справочников (необходимые для обеспечения целостности ссылок и автономной работы Заказчика) и все транзакционные данные (пользователи, доступы, анкеты).
   - **Синхронизация и Инициализация (Deployment Sequence)**: 
     Обновление локальных справочников в базах заказчиков производится **строго на этапе первичного развертывания** (до регистрации реальных пользователей).
     1. Запуск приложения и создание пустой структуры БД (`schema.sql`).
     2. Синхронизация (`npm run migrate:refdb` или API POST `/api/admin/import`), которая полностью очищает (`TRUNCATE CASCADE`) локальные справочники и заполняет их свежими данными из `refdb`, генерируя новые внутренние ключи (`id`).
     3. Создание пользователей-респондентов администратором системы и выдача им прав.
     Эта модель исключает проблемы со ссылочной целостностью в рабочих данных (анкетах), так как структура справочников фиксируется до начала работы людей.

## 5. Архитектурные инварианты и решения (ADR)

Для поддержания качества и надежности кода зафиксированы следующие правила:

1. **Модульность бэкенда**:
   Логика API разделяется по доменам (отдельные плагины Fastify: авторизация, справочники, ответы пользователей, процессы). Контроллер `server.js` должен быть точкой сборки, а не "свалкой" всех роутов.
2. **Управление схемой БД (Migrations)**:
   Отказ от механизма автоматического создания таблиц "на лету" (Auto-migrate на старте сервера). Развитие базы данных должно происходить только через скрипты направленных миграций (например, `node-pg-migrate`). Это обеспечивает версионирование структуры БД и безопасные деплои.
3. **Бизнес-логика в коде приложения**:
   Оркестрация сложных процессов (например, генерация анкет респондентов) реализуется на слое бэкенда (Node.js) в рамках транзакций. База данных отвечает за хранение, выборку и консистентность на уровне ограничений (constraints, foreign keys), а не за бизнес-процессы внутри хранимых процедур.
4. **Security и принцип недоверия**:
   API никогда не доверяет данным с клиента. Любое действие (просмотр, изменение данных) верифицируется на сервере: проверяется валидность JWT, роль пользователя (`admin` / `user`) и jego права доступа к конкретным данным (scope).
5. **Предотвращение SQL-инъекций**:
   Строго запрещена конкатенация строк в SQL-запросах. Все переменные передаются исключительно через массивы параметров драйвера `pg` (`$1, $2`).
